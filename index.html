Section 01 · Foundation

Start Here — What Problem Are We Solving?

Before touching a single line of code, you need to understand why this problem matters. Let's build the mental model from scratch.

🧠 What is an "embedding"?

AI models can't understand raw text or images. So they convert everything into a list of numbers — called a vector or embedding.

Example: "The cat sat on a mat"
→ [0.12, -0.56, 0.88, 0.34, -0.21, 0.67, ...]
960 numbers representing the "meaning" of this sentence

Two similar sentences will produce vectors that are close together in space. Two unrelated sentences will be far apart.

🔍 What is similarity search?

Given a query (e.g. "Where is the cat?"), find the most similar items from a database of millions of stored embeddings.

Real-world uses:
● ChatGPT searching its memory (RAG)
● Google Images — "find similar photos"
● Spotify — "users who like this song also like…"
● Amazon — product recommendations
● Fraud detection — find suspicious patterns

The Core Problem: Speed vs Scale

Here's the challenge. Imagine you have 1 million product embeddings, each with 960 dimensions (numbers). A customer searches for something. How do you find the 10 most similar products?

❌ The naive approach (what everyone tries first)

Compare the query against every single product. 1,000,000 × 960 = 960 million comparisons per search. On a modern server, this takes about 0.6 seconds. For an e-commerce site with 1000 searches per second, that's completely unusable.

✅ What FAISS does

FAISS (Facebook AI Similarity Search) is a library with 9+ clever algorithms that find the approximately nearest neighbors in milliseconds. It trades a tiny bit of accuracy for massive speed gains — and for most real applications, finding the 3rd-closest item instead of the 1st-closest is perfectly acceptable.

🎯

Exact Search (kNN)

Always finds the true nearest neighbor. 100% accurate. But must check every single vector. Unusable at scale.

⚡

Approximate (ANN)

Finds a neighbor that's close enough. Might miss the absolute best match, but 1000× faster. Used in all production AI systems.

📊

Recall@10

The key accuracy metric: "Of the 10 true nearest neighbors, how many did the algorithm actually find?" 1.0 = perfect. 0.8 = found 8 out of 10.

Core Code — The Benchmarking Function

Every algorithm goes through this same function. The key line is idx.search(queries, 10) — ask the index "find the 10 nearest neighbors for each query".

def run_benchmarks(base, queries, ground_truth):
    # Step 1: nlist = number of clusters = sqrt(N), a standard rule of thumb
    nlist = int(np.sqrt(base.shape[0]))  # e.g. sqrt(1,000,000) = 1000 clusters

    # Step 2: Build the index (varies by type)
    idx = faiss.IndexFlatL2(d)  # example: brute-force index
    idx.add(base)              # add all 1M vectors to the index

    # Step 3: Search — this is the key operation we're benchmarking
    start = time.perf_counter()
    distances, neighbor_ids = idx.search(queries, 10)  # find top-10 for each query
    elapsed = time.perf_counter() - start

    # Step 4: Calculate how fast (QPS) and how accurate (Recall@10)
    qps = len(queries) / elapsed
    # Recall@10 = overlap between our results and the true nearest neighbors
    recall = mean(overlap(neighbor_ids, ground_truth))
Section 03 · The Algorithms

The 9 Indexing Methods — Fully Explained

Each algorithm takes a completely different approach to the same problem. Here's what each one does and why, with no prior knowledge assumed.

EXACT SEARCH

1. IndexFlatL2 — The Gold Standard Baseline

What it does

Compares your query against every single vector in the database using Euclidean (straight-line) distance. No shortcuts, no approximations. "L2" means it uses the L2 (Euclidean) distance formula: √(Σ(aᵢ−bᵢ)²).

Real analogy

Finding the nearest hospital by checking the distance to every hospital in the country one by one. Guaranteed to find the closest — but very slow.

Used as the ground truth. Every other method is compared against this. Its recall is nearly 1.0 by definition.

INVERTED FILE

2. IndexIVFFlat — Partition and Conquer

What it does

Uses k-means clustering to divide all vectors into Voronoi cells (like territories). Each vector "belongs" to its nearest cluster centroid. When searching, it only looks inside the nprobe closest cells — not the whole dataset.

Real analogy

Finding the nearest hospital in a city divided into districts. Step 1: find which district you're in. Step 2: search only that district (and maybe a couple neighboring ones). Much faster than checking the whole country!

nprobe is the key tuning knob. nprobe=1 → fastest but might miss the answer (the "edge problem"). nprobe=50 → slower but more accurate.

The Edge Problem

If your query sits near the border between two cells, the true nearest neighbor might be in the adjacent cell — but you didn't search it. This is why recall is ~70% not 100%. Increasing nprobe helps.

QUANTIZATION

3. IVFPQ — IVF + Product Quantization (Memory Saver)

What it does — two stages

Stage 1 (IVF): Same clustering as above — divide into Voronoi cells.
Stage 2 (PQ): Instead of storing full float32 vectors, compress each vector into tiny integer codes.

PQ works by splitting each 960-dimension vector into 8 sub-vectors of 120 dimensions each. Each sub-vector is mapped to the nearest centroid ID — just one byte. So 960 floats (3,840 bytes) becomes 8 bytes. 480× smaller!

Real analogy

Instead of remembering a person's exact face (high-res photo = 10 MB), you describe them using 8 checkboxes: tall/short, dark hair/light hair, etc. (8 bytes). You lose some detail but can store millions of descriptions cheaply.

⚠️ The Dimensionality Problem

On GIST-960 (high dimensions), the recall collapses to just 10%. This is the "Curse of Dimensionality" — in high-dimensional space, the compression loses critical information. This is the most dramatic finding in the paper!

On DEEP-96 (low dimensions), it performs much better: 6.4% recall but with 1.7 million QPS.

SCALAR QUANT

4. IVFSQ — Simpler Compression (The Reliable Middle Ground)

What it does

Uses IVF clustering, but compresses vectors using Scalar Quantization (SQ): converts each 32-bit float into an 8-bit integer by mapping the full range to 256 discrete values. A fixed 4× compression ratio.

Real analogy

Like rounding prices to the nearest dollar. A $9.99 item becomes $10. You lose a tiny bit of precision but the compression is predictable and reliable. Much simpler than PQ.

OPTIMIZED PQ

5. IVFOPQ — PQ Fixed for High Dimensions (The Smart Fix)

What it does

IVFPQ fails in high dimensions because some dimensions carry much more information than others (unequal variance). IVFOPQ fixes this by learning an orthogonal rotation matrix R that spreads the information evenly across all dimensions before quantizing.

The math:
x_rotated = x × R
R is a learned matrix that re-balances variance across all sub-spaces, so PQ works equally well everywhere.

GRAPH-BASED

6. IndexHNSWFlat — The Speed Champion (Multi-Layer Graph)

What it does

Builds a multi-layer graph where each node (vector) is connected to its nearest neighbors. The top layers have long-range connections for fast global navigation. Lower layers have dense local connections for precise search.

Search starts at the top layer, finds the rough area, then descends layer by layer to refine — like going from a world map to a city map to a street map.

Real analogy

GPS navigation: first find which continent, then country, then city, then street. Each layer zooms in. Much faster than checking every road in the world.

HASHING

7. IndexLSH — Locality Sensitive Hashing (Fast but Weak)

What it does

Projects vectors into binary hash codes (strings of 0s and 1s) using random hyperplanes. Similar vectors are designed to produce similar hash codes — so you only compare vectors with matching or near-matching hashes.

Real analogy

Sorting library books by the first letter of the title. Fast to find the right shelf — but similar books (e.g. "Algorithm Design" and "Algorithms Explained") might end up in different sections because hashing is lossy.

Understanding Every Chart — Plain English

Fig 1 Recall-Throughput Pareto Frontier

This is the most important chart in the paper. It plots every algorithm on two axes: how accurate (X) vs how fast (Y).

How to read this chart

  • X-axis (Recall@10): How accurate. Further right = more accurate. 1.0 = perfect.
  • Y-axis (QPS, log scale): How fast. Higher = more searches per second. The Y-axis is logarithmic — each grid line is 10× bigger, not just bigger by a fixed amount.
  • The ideal algorithm would sit in the top-right corner (fast AND accurate). No algorithm achieves this — it's the fundamental trade-off.
  • Solid line = GIST dataset, Dashed line = DEEP dataset. Points connected by lines show different nprobe settings for the same algorithm.
  • "Pareto frontier" means the algorithms that aren't dominated by any other — no other algorithm is both faster AND more accurate than them.

What you'll see: The big cluster on the left

Most algorithms on GIST are clustered near Recall≈0.05–0.2 (left side). This shows that high dimensionality is brutal. Even algorithms that work well on DEEP-96 collapse on GIST-960. Notice IVFPQ (green) is very fast (top of Y axis) but has terrible recall on GIST (far left on X axis). FlatL2 (pink) sits alone at Recall=1.0 on the right — perfect accuracy but lower speed.

Section 07 · Research Paper

Deep Dive: The Research Paper Analysis

The paper "Dimensionality and Scalability in Approximate Nearest Neighbor Search: A 9-Method FAISS Benchmark" makes 6 major empirical findings. Here's what each one means and why it matters.

🏆

Finding 1: HNSW defines the upper boundary of accuracy-speed trade-off

On the GIST-960 dataset, HNSW achieves 83.1% Recall@10 with 625 QPS. No other algorithm gets above 83% recall while remaining faster than brute-force. This makes HNSW the "benchmark king" — but it comes at severe cost. On GIST-1M: 10,647 seconds to build and 8,550 MB RAM.

What this means for you

If you're building a production AI system and you have money for server RAM, HNSW is the right choice for user-facing search (semantic search, RAG). If you're building on a budget or with streaming data, look at the quantization methods instead.

💥

Finding 2: The "Curse of Dimensionality" destroys standard IVFPQ on GIST

This is the most dramatic finding. Standard IVFPQ achieves only 10.1% Recall@10 on GIST-960 at 1M scale. Why? Because in 960 dimensions, information is distributed unevenly — some dimensions carry enormous variance while others carry almost none. When PQ splits the vector into 8 equal sub-spaces, it forces equal importance on unequal parts. The result: catastrophic compression loss.

GIST-960 (high-D) — collapse

IVFPQ recall: 10.1%. The compression throws away most of the signal.

DEEP-96 (low-D) — works fine

IVFPQ recall: 6.4% (low recall here is a metric mismatch, not compression failure).

🔑

Finding 3: IVFOPQ partially recovers from dimensionality collapse — nearly 2×

By applying a learned rotation matrix R before quantization, IVFOPQ achieves 18.7% recall on GIST-1M vs IVFPQ's 10.1% — almost double — while using essentially the same ~4,631 MB of memory. This proves that the PQ failure was due to unequal variance, not an inherent limitation of quantization.

The trade-off: build time

IVFOPQ took 947 seconds (16 minutes) to build on GIST-1M vs 11 seconds for IVFPQ. The rotation matrix learning is computationally expensive. But for a one-time index build on a system that searches billions of times afterward, this is an excellent trade-off.

📈

Finding 4: HNSW scales super-linearly on GIST — a deployment blocker

Going from 100K to 1M vectors (10× growth) should ideally take 10× longer to build. HNSW on GIST-960 took 18.9× longer — super-linear. In contrast, IVFFlat took 4.5× longer (sub-linear). IVFOPQ took only 1.3× longer (nearly constant).

The cause: building proximity links in 960-D space requires comparing each new vector to many potential neighbors. In high dimensions, finding these neighbors is computationally hard because of distance concentration — all points tend to be at similar distances from each other, making it hard to identify true neighbors efficiently.

⚖️

Finding 5: IVFPQ and HNSW occupy opposite deployment poles

These two algorithms represent the extreme ends of a deployment spectrum:

IVFPQ — Resource Minimalist
  • Memory: ~4,618 MB (no extra)
  • Build: 11 seconds
  • QPS: 129,320
  • Recall: 10.1% on GIST
  • Best for: memory-constrained, fast QPS
HNSW — Performance Maximalist
  • Memory: 8,550 MB (+85% extra)
  • Build: 10,647 seconds
  • QPS: 625
  • Recall: 83.1% on GIST
  • Best for: high accuracy, premium infra
🧩

Finding 6: The radar charts reveal algorithm "specialization" under stress

On DEEP-96, the radar chart shows balanced polygons — most algorithms perform reasonably across all 4 metrics. But on GIST-1M, the profiles "polarize": IVFPQ becomes a pure speed specialist (massive QPS, terrible recall), FlatL2 becomes a pure accuracy specialist (perfect recall, no speed), and HNSW sits in an uncomfortable middle — good recall, but terrible build and memory efficiency. The data proves there is no free lunch in high-dimensional similarity search.

Methodology: Strengths & Limitations

✅ Strengths

  • 9 architectures — most comprehensive FAISS comparison in student research
  • Two structurally different datasets (validates generalizability)
  • Two scales (100K and 1M) — captures scaling behavior
  • 5 metrics — multi-dimensional view, not just accuracy
  • GPU-accelerated on NVIDIA T4 — realistic production conditions
  • psutil RSS tracking — honest system memory measurement
  • Gaussian noise augmentation — scientifically sound scaling approach
  • Multiple nprobe settings for IVF methods — shows the tuning landscape

⚠️ Limitations

  • Data augmentation (noise tiling) is synthetic — real 1M distributions have richer structure
  • Google Colab has variable GPU allocation — build times may not be perfectly reproducible
  • Fixed nprobe=10 for most experiments — different settings might change method rankings
  • Only two datasets — adding SIFT-1B or Glove would strengthen claims
  • HNSW hyperparameters (M=32, efSearch=64) — different M could change QPS/recall balance
  • Memory measurement includes OS baseline (~4,600 MB) — makes small differences invisible
  • DEEP-96 metric mismatch (angular GT vs L2 search) creates misleading recall numbers

Evidence-Based Production Guidelines

⚡ Latency-Critical
HNSW

For semantic search, LLM RAG, real-time recommendations where millisecond responses matter. Verified: 625 QPS at 83% recall on 1M GIST vectors. Plan for 8.5 GB RAM per million vectors.

IVFPQ or IVFOPQ

For billion-scale archives, edge deployment, or when RAM is limited. IVFOPQ doubles IVFPQ's recall (18.7% vs 10.1%) on high-D data with nearly the same memory footprint — 947s build is worth it.

🎯 High-Dimensional
IVFOPQ mandatory

For modern LLM embeddings (512-D, 1536-D, 3072-D), standard PQ is mathematically broken. The orthogonal rotation learned by OPQ is the only way to prevent recall collapse without switching to HNSW.
